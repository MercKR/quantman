# 블록 IR 구현 명세 (Block IR Implementation Spec)

> **문서 위상.** 이 문서는 [`atomic_block_spec_v2.md`](../../../atomic_block_spec_v2.md)(목표 *비전*)의 짝이 되는 *구현 명세*다.
> 비전 문서가 "무엇을 만들 것인가"를 정의한다면, 이 문서는 "그것을 어떤 자료구조·모듈·단계로 구현할 것인가"를 정의한다.
> 비전 문서의 모든 §은 본 문서 끝의 **추적표(§15)** 에서 구현 위치와 1:1로 대응된다.
>
> **전제.** 현재 사용자는 전원 알파 테스터로 마이그레이션 제약이 없다. 따라서 하위호환 어댑터 없이
> **백지(clean slate) 재설계**한다. 임기응변식 기능 덧대기를 금하고, 가장 단순·최적화된 구조를 택한다.
>
> **이번 범위.** 전략 *정의 구조* + *백테스트/분석* 엔진. 라이브 실행(trader/preview)으로의 브리지는 Phase 3.
> NL→블록 컴파일러는 최종(Phase 3).

## 1. 설계 목표와 원칙

### 1.1 3단계 목표

1. **자유도** — 문장형 빈칸은 유지하되, 고급 퀀트 전략(횡단 순위·그룹 중립화·시계열 관계·페어)을 코드 없이 표현.
2. **노코드 분석 구조** — 비전 문서의 atomic block 시스템 전체(데이터→신호→포지션→성과→시뮬 + 펼침·비교·무결성).
3. **자연어 컴파일러** — 자연어 분석 요청 → 블록 트리 자동 생성 → 규칙 기반 분석.

### 1.2 설계 철학 (비전 §0 — 본 명세의 불변 제약)

| 원칙 | 본 명세에서의 강제 수단 |
|---|---|
| 원자성 | 블록 카탈로그(§3)는 "자연어 한 구절로 완결되는 최소 단위"만 등록 |
| MECE | 6축 분류(§3)가 같은 층위에서 배타·전수. 중복 블록 등록 금지 |
| 재사용 우선 | 새 기능은 새 블록이 아니라 **기존 블록 조합**으로 표현(가지 빈칸) |
| 무결성 최우선 | `integrity.py`(§6)가 평가·시뮬을 관통. 편의보다 항상 우선(규칙 충돌 §5.3) |
| retail 친화 | 규칙5 기본값(§5.2·§13)으로 빈칸 미완성에도 동작 |

## 2. 핵심 아키텍처 — 통합 타입드 블록 IR

### 2.1 왜 통합 IR인가

현재 시스템엔 전략 문법이 **셋으로 분산**돼 있다:

- `analysis.py`의 조건 문법(Condition/Operand) — 문장형, 스칼라 잎, long-only 트리거
- `expression_parser.py`의 alpha 수식 — **코드 문자열**, 횡단·시계열 연산 풍부, long/short
- `screener.py`의 screener spec — 별도 필드·연산자, 라이브/백테스트 이중 구현

세 문법은 같은 "데이터에 연산을 걸어 신호를 만든다"는 일을 서로 다른 표현으로 한다. **통합 IR은 이 셋을 하나의 재귀 트리(Node)로 대체한다.** 그 결과 평가기·UI·LLM이 단일 스키마를 공유한다.

### 2.2 타입 시스템 (비전 §1.2 값 타입 + §3.1 형태축)

모든 블록은 **값 타입**과 **데이터 형태**를 태그한다. 이게 메타규칙(§5)의 검증 근거다.

**값 타입 6종** (`ValueType`):

| 타입 | 의미 | 예 |
|---|---|---|
| `score` | 실수 패널(점수) | 순위 0.7, 수익률 3% |
| `condition` | 참/거짓 패널 | "유가 80 미만" |
| `scalar` | 단일 스칼라 | Sharpe 1.5 |
| `label` | 분류 라벨 패널 | 업종='반도체', 변동성='고' |
| `distribution` | 스칼라 집합 | 수익률 히스토그램 |
| `resultset` | 백테스트 결과 묶음 | 펼침(§8) 출력 |

**데이터 형태 5종** (`Shape`):

| 형태 | 의미 | 적용 제약 |
|---|---|---|
| `panel` | 종목×날짜 행렬 | 모든 블록 가능 |
| `series` | 시장 단일값(날짜×1) | 횡단·그룹 연산 무의미(규칙2 차단) |
| `vector` | 한 칸 다값(뉴스·옵션체인) | 압축(축5) 선행 강제 |
| `grouplabel` | 업종·섹터 분류 | 그룹 연산 기준으로만 |
| `static` | 거의 불변(상장일·국가) | 주로 대상 종목 선정 |

### 2.3 Node / Leaf 스키마

```python
# core/quant_core/blocks/node.py
class Node(BaseModel):
    op: str                          # 블록 종류 (catalog 등록 키)
    inputs: dict[str, "Node | Leaf"] # 가지 빈칸 — 슬롯명 → 하위 블록(재귀)
    params: dict[str, Any] = {}      # 잎 빈칸 — window·option·groupkey·value 등
    # type·shape는 평가/검증 시 catalog의 타입규칙으로 추론(저장 안 함 — 단일 진실)

class Leaf(BaseModel):
    kind: Literal["data", "constant", "period", "groupkey", "option", "condition", "percent"]
    ref: Optional[str] = None        # kind="data": 데이터 키(예: "Close", "VIX.Close", "__SELF__")
    value: Optional[Any] = None      # constant/period/option/percent
```

**현재와의 결정적 차이**: 기존 `Operand`는 스칼라 잎만 가능했다. `Node.inputs`는 **다른 Node를 받는다**(비전 §1.3 가지 빈칸). 이 한 가지가 중첩 전략을 푼다.

### 2.4 잎 빈칸 / 가지 빈칸 (비전 §1.3)

- **잎 빈칸**(드롭다운/입력, 더 안 쪼개짐): `data`(종가·거래량…)·`period`(20일)·`groupkey`(업종)·`option`(평균/합 등 열거)·`condition`(기성 참/거짓)·`percent`(비율).
- **가지 빈칸**(블록 문장이 통째로 들어감): `signal`. 여기에 또 다른 블록을 끼우는 것이 "중첩". 복잡한 전략은 전부 이 중첩으로 만든다.

### 2.5 통합 예시 — 두 기존 문법이 같은 IR로

기존 조건 `[이 종목] RSI(14) < 30`:
```json
{"op":"compare","params":{"op":"<"},
 "inputs":{"left":{"op":"data","inputs":{},"params":{"ref":"__SELF__.rsi_14"}},
           "right":{"kind":"constant","value":30}}}   // → type: condition
```
기존 alpha `rank(momentum_12_1m)`:
```json
{"op":"rank","inputs":{"signal":{"op":"data","inputs":{},"params":{"ref":"momentum_12_1m"}}}}
// → type: score
```
**현재는 불가능하나 새 IR로 가능한 고급 전략** — "전체 종목 중 〈최근 10일 종가·거래량 동행성〉 순위 상위":
```json
{"op":"rank","inputs":{"signal":
  {"op":"ts_corr","params":{"window":10},
   "inputs":{"a":{"op":"data","params":{"ref":"Close"}},
             "b":{"op":"data","params":{"ref":"Volume"}}}}}}
// → type: score (가지 빈칸에 ts_corr 블록이 중첩)
```

## 3. 블록 카탈로그 (비전 §3.2 6축 — 전수)

각 블록은 `catalog.py`에 `op → {입력 슬롯·타입규칙·params·출력타입·평가함수}`로 등록한다.
**엔진 상태**: ✅=현재 코드 존재(승격) · 🆕=신규 구현.

### 축1. 산술·논리 (한 칸 연산)

| op | 하는 일 | 입력→출력 | 상태 |
|---|---|---|---|
| `unary` | log·abs·sign·sqrt (option) | score→score | ✅ |
| `binary` | ＋−×÷ (option) | score,score→score | ✅ |
| `compare` | >,>=,<,<=,between (op) | score,(score/const)→condition | ✅ |
| `select` | 조건 참→A 아니면 B (조건부제어) | condition,score,score→score | ✅(if_else) |
| `logic` | AND/OR (logic) | condition,condition→condition | ✅ |
| `cross` | 상향/하향 돌파 | data,(value/signal)→condition | ✅ |
| `bucket` | 경계값으로 구간 라벨 부여 | score,경계리스트→label | 🆕 |

### 축2a. 단일 시계열 (한 종목 과거)

| op | 하는 일 | 상태 |
|---|---|---|
| `ts_mean` `ts_sum` | 최근 N일 평균·합 | ✅ |
| `ts_std` | 최근 N일 산포(변동성) | ✅ |
| `ts_rank` | 윈도우 내 위치 | ✅ |
| `ts_delta` `ts_delay` | 변화량·N일 전 값 | ✅ |
| `ts_argmax` `ts_argmin` | 최고/최저 시점까지 거리 | ✅ |
| `ts_decay` | 시간가중 평활(가까운 날 비중↑) | ✅ |
| `ts_halflife` `ts_autocorr` | **시계열 동역학** — 평균회귀 반감기·지속성·자기상관 | 🆕 *(보강①)* |

### 축2b. 시계열 간 관계 (두 변수)

| op | 하는 일 | 상태 |
|---|---|---|
| `ts_corr` | 최근 N일 동행성(상관) | 🆕 |
| `ts_regression` | 선형적합 — 베타 또는 잔차(option) | 🆕 |
| `orthogonalize` | 한 신호에서 다른 신호 방향 성분 제거 | 🆕 |

### 축3. 횡단 (같은 날 전체 종목)

| op | 하는 일 | 상태 |
|---|---|---|
| `rank` | 그날 전체 순위(0~1) | ✅ |
| `zscore` | 그날 평균·표준편차 환산 | ✅ |
| `scale` | 비중 합=1 포지션 스케일 | ✅ |
| `cs_dispersion` | 그날 종목값 산포 | 🆕 |

### 축4. 그룹 (같은 업종 내)

| op | 하는 일 | 상태 |
|---|---|---|
| `group_rank` | 그룹 내 순위 | 🆕 |
| `group_aggregate` | 그룹별 평균·합을 그룹에 부여 | 🆕 |
| `group_neutralize` | 그룹 평균 차감(그룹 효과 제거) | ✅(섹터 데이터 필요 §7.1) |

### 축5. 벡터 (한 칸 다값 압축)

| op | 하는 일 | 상태 |
|---|---|---|
| `vec_stat` | 하루치 여러 값 평균·합·최대 | 🆕 |
| `vec_count` | 이벤트 발생 건수 | 🆕 |
| `vec_magnitude` | 여러 값 전체 크기 | 🆕 |

### 목적 태그 (방향 무관, 어느 블록에든 부착)

`normalize`(정규화) · `neutralize`(중립화) · `decay`(평활화) · `select`(조건부제어 — 조건 충족 시만 활성) · `winsorize`(이상치 처리). *(조건부제어는 hump가 아니라 select로 매핑 — 보강 검토 반영)*

> **MECE 점검**: 약 30블록 중 ~20블록은 현재 `expression_parser.py`/`analysis.py`에 평가 코드가 존재한다. 재설계의 큰 비중은 *새 엔진 작성이 아니라* 흩어진 연산을 타입드 블록으로 통합·문장 노출하는 일이다.

## 4. 평가기 (evaluate)

```python
# core/quant_core/blocks/evaluate.py
def evaluate(node: Node, dataset: Dataset, ctx: EvalContext) -> Typed:
    """블록 트리를 순회하며 타입드 패널/시리즈/스칼라를 산출."""
```

- 매수 조건 = `type:condition` 루트 Node → 현재 `build_signal_mask` 대체
- 알파 가중치 = `type:score` 루트 Node → 현재 `ExpressionParser.evaluate` 대체
- 스크리너 = `condition` 필터 + `rank` 정렬 Node → screener 이중구현 통합

`EvalContext`는 `current_symbol`([이 종목] 치환), `delay`(무결성), `as_of`(PIT)를 운반한다.

## 5. 메타규칙 엔진 (비전 §8) — `validate.py`

블록 조립 시 잘못된 결합을 막고 합리적 결과를 보장한다.

### 5.1 규칙 0~5

| 규칙 | 내용 | 구현 |
|---|---|---|
| **규칙0** 데이터 가용성 선검사 | 블록 평가 전 데이터 존재 확인. 없으면 "해당 데이터 없음" | catalog data-ref 사전 lookup *(보강④)* |
| **규칙1** 타입 호환성 | 빈칸별 허용 타입 강제(`signal`=score, `condition`빈칸=condition만) | catalog 타입규칙 |
| **규칙2** 형태 호환성 | `series`(유가)엔 횡단·그룹 비활성, `vector`엔 압축 선강제 | Shape 검사 |
| **규칙3** 조립 순서 | 원재료블록(시계열·벡터) 안쪽, 신호받는블록(횡단·평활) 바깥쪽 + 포지션 짝 제약 | 토폴로지 검사 |
| **규칙4** 무결성 강제 | §6 데이터·파라미터 시점 무결성 자동 적용 | integrity.py 위임 |
| **규칙5** 완결성·기본값 | 빈칸 미완성에도 합리적 기본값(기간 20일·동일가중·시뮬 기본)으로 동작 | §13 기본값 정책 *(보강⑤)* |

### 5.2 retail 기본값 정책 (규칙5)

기간 N=20, 크기=동일가중, 비용=시장 기본, delay=1, NaN=제외, PIT=켬, 종목당상한=10%, 초기자본=1억·레버리지=1배.

### 5.3 충돌 우선순위 *(보강④)*

**무결성(규칙4) > 정합성(규칙1·2·3) > 완결성(규칙5).**
무결성 경고는 항상 표시. 타입·형태·순서 오류는 기본값으로 덮지 않고 거부. 기본값 주입이 무결성을 해치면 채우지 않고 사용자에게 묻는다.

## 6. 무결성 게이트 (비전 §6) — `integrity.py`

시스템의 최대 차별점. 자연어 요청을 신뢰성 있는 실행으로 자동 번역한다.

### 6.1 데이터 시점 무결성 — "그 시점에 알 수 있던 값만"

- **시점지연(Delay)**: 돌파·급락은 종가 확정 후에야 안다 → 익일 진입 강제(기본 1일).
- **시간정합성(PIT)**: 사후 수정된 재무·추정치가 아닌 *당시 발표값* 사용. 데이터에 발표일 꼬리표 필요.
- **생존편향 방지**: 지수 구성은 시간에 따라 바뀜 → 각 시점에 실제 속했던 종목 사용(멤버십 이력 데이터 필요).

### 6.2 파라미터 시점 무결성 — "기준값을 미래로 정하지 않았나"

- 분할 임계값·최적화 비중·바스켓 가중치(β)를 **전체 기간으로 정한 뒤 과거 적용 금지** → 각 시점까지 데이터로만(롤링) 산정.
- 무작위 k-fold 요청은 **시계열 k-fold로 자동 치환**(시간 순서 보존 + 구간 사이 공백).

## 7. 1회 백테스트 — 5단계 (비전 §3) — `engine/backtest.py`

### 7.1 데이터층 (비전 §3.1)

- **형태축 5종**: §2.2 Shape와 동일.
- **내용축 9종** *(보강②)* — 데이터 피커 네비게이션 + 데이터 갭 분류:
  가격·거래(PV) · 재무 · **추정치**(컨센서스·EPS revision) · **뉴스·센티먼트** · **이벤트·수급** · 거시 · **레퍼런스·상태**(상장·관리종목) · **대체데이터** · **배당·total return**.
  현재 보유: PV·재무(flat lag)·거시. **부재**: 추정치·뉴스·이벤트수급·레퍼런스/상태·대체·배당 → Phase 3 점진 보완.
- **PIT**: §6.1. 현재 펀더멘털은 flat +45d로 *진짜 PIT 아님* → Phase 2~3 발표일 기반 전환.

### 7.2 신호층

= §3 블록 트리. `type:score`(점수화) 또는 `type:condition`(트리거).

### 7.3 포지션층 (비전 §3.3) — `engine/position.py`

신호를 실제 매매로 번역. 네 부품 독립 조합:

| 부품 | 선택지 | 상태 |
|---|---|---|
| ① 방향 | 롱 / **숏** / 롱숏중립 | 숏·중립 🆕(현재 long-only) |
| ② 크기 | 동일가중 / 신호비례 / **변동성역가중** / **고정위험** / **켈리** | 뒤 3종 🆕 *(보강③)* |
| ③ 진입 | 상시 / 조건충족시 / 정기리밸런싱 | ✅ |
| ④ 청산 | 매일갱신 / N일후 / 조건시 / 손익절 | ✅ |

- **전역 오버레이**: 변동성타게팅 · 종목당상한 · 턴오버억제.
- **짝 제약**(규칙3): 상시진입 ⇒ 매일갱신 청산만. 고정위험·켈리는 손절폭·승률 입력 필요.
- **사이징 통일**: 현재 백테스트는 `sizing_mode`를 무시(all-in/amount_pct)하고 라이브만 사용 → **백테스트·라이브 동일 사이징 엔진**으로 통일(Phase 2).

### 7.4 성과층 (비전 §3.4) — `engine/performance.py`

- **산출물 3종**: 일별 수익률 시계열 · 트레이드 목록 · 포지션 시계열.
- **집계 블록**: 평균 · 표준편차(하방 선택) · 누적성장(연율화) · 최대낙폭 · 분위수 · 개수·비율 · 회귀계수(β) · **초과수익(CAR)** · **분포 출력**(type:distribution).
- **조합 프리셋**: Sharpe · Sortino · CAGR · Calmar · MDD · VaR · CVaR · 승률 · Profit Factor · 베타. *(현재 ~8지표뿐 → 전량 보강)*

### 7.5 시뮬레이션층 (비전 §3.5)

## 8. 펼침층 SWEEP (비전 §4) — `engine/sweep.py`

전략 고정, 한 축을 변화시키며 반복 실행 → `resultset`.

- **조건축**: 한 번 실행 후 결과를 **사후** 조건 라벨로 분할(횡단/이벤트/시간구간). 도구: `bucket` · 국면 라벨링(VIX 저/중/고) · 달력 라벨링(요일·월중).
- **파라미터·자산축**: 설정·대상을 바꿔 **재실행**(거래비용 0·5·10bp, BTC·ETH·SOL).
- **시간축**: 이벤트 기준 재정렬(이벤트 스터디 — 발표일=0) + 롤링 기간 분할.

> 조건축=사후 분할, 파라미터축=사전 반복.

## 9. 비교·검증층 (비전 §5) — `engine/compare.py`

`resultset`을 받아 차이가 유의한지 판정.

- **비교 도구**: 이표본 차이검정 · **다표본 차이검정**(셋 이상/추세) · **predictor 설명력 비교** · **조건부 꼬리 비교** *(보강⑥ — 다표본·predictor·조건부꼬리)*.
- **유의성 검정**: 부트스트랩 · **몬테카를로**(트레이드 순서 셔플) · t-test · 워크포워드 유의성 *(보강⑥ — MC)*.
- **다중검정**: 같은 데이터 반복 검증 시 우연 유의 문제. 보정 여부는 향후 결정(비전과 동일 보류).

## 10. 거래단위 일반화 (비전 §7)

기본 단위는 개별 종목. 가중 결합을 하나의 **바스켓**으로 묶어 관계를 거래.

- **페어트레이딩**: "KO 매수 +(β) PEP 매도" = 가중치 (+1,−β) 바스켓. β=`ts_regression`(축2b). 바스켓 위에서 포지션 4부품 그대로 작동.
- **헤지 오버레이**: "알파 포트 + 지수선물 매도" 바스켓 + 전역 조건(포트 베타>0.3).
- **무결성**: β도 파라미터 시점 무결성으로 롤링 추정(§6.2).

### 10.1 명시적 범위 밖 *(보강⑧, 비전 P3 보류와 동일)*

- **포트폴리오 최적화**(평균-분산·리스크 패리티) — 가중치가 사전 미정, 별도 QP 솔버 필요. 본 일반화로 안 풀림.
- **옵션 등 비선형 손익 상품** — 같은 이유로 현재 범위 밖.

## 11. NL → IR 컴파일러 (비전 §9) — 최종(Phase 3)

자연어 요청 처리 8단계. **§5 메타규칙·§6 무결성을 그대로 재사용**한다(그래서 Phase 0~2에서 먼저 굳힌다):

1. 데이터 존재 확인(규칙0)
2. 비교·반복 축 감지 → 펼침층 경유 여부
3. 데이터→신호→포지션→성과 블록 매핑
4. 펼침이면 결과세트 → 비교·검증
5. 타입·형태·순서 검사(규칙1·2·3)
6. 무결성 검사·안전판(규칙4)
7. 빈칸 기본값 주입(규칙5)
8. 통과 시 실행, 막히면 어느 단계서 막혔는지 설명

LLM은 자유 텍스트가 아니라 **타입드 Node 트리**를 생성하므로 검증 가능하고 안전하다.

## 14. 단계별 로드맵 (비전 §10 P0~P3 정렬)

> 비전 문서의 우선순위에 1:1 정렬. 각 Phase는 측정 가능한 검증 게이트를 가진다.

### Phase 0 — IR 기반 + 무결성 (비전 P0)
- `blocks/` 전체: types · node · catalog · evaluate · validate(규칙0·1·2·3·5) · integrity(delay·PIT·생존편향 플래그)
- 기존 grammar/expression 연산을 블록으로 이식
- **검증**: 블록별 타입추론·평가 단위테스트 / **기존 `golden_backtest`를 새 평가기로 동일 재현**(재설계가 결과 안 깸 증명)

### Phase 1 — 신호 자유도 + 펼침 조건축 + 비교층 (비전 P1)
- 문장 빌더가 **가지 빈칸 + 횡단·그룹·시계열동역학·교차·dispersion·bucket** 렌더
- **섹터/산업 분류 데이터 도입**(group 연산 실효화)
- **펼침 조건축** + **비교층 4종 + 분포 출력** + 초과수익(CAR)
- 백테스트가 Node IR 직접 평가
- **검증**: 브라우저로 "모멘텀 순위 상위 N"·"변동성 高/低 비교" 조립·실행 / 통계검정 수치 대조

### Phase 2 — 펼침 확장 + 포지션·성과·시뮬 (비전 P2)
- 파라미터/자산축 · 시간축·이벤트 스터디
- 포지션 4부품(크기 5종·짝 제약·오버레이) + **백테스트/라이브 사이징 통일**
- 성과 전체(Sortino·Calmar·VaR·CVaR·PF·β) + 시뮬(레버리지·Region·기간분할 워크포워드/k-fold)
- **검증**: 이벤트 스터디 정렬·민감도 그리드 / paper 시나리오로 사이징 일치 확인

### Phase 3 — 데이터 심화 + 페어/헤지 + NL 컴파일러 (비전 P2~P3 + tier3)
- 데이터 5범주 점진(추정치·뉴스·이벤트·수급·배당) + 진짜 PIT
- 페어·헤지(§10) + **NL→IR 컴파일러**(§11) + **라이브 실행 IR 브리지**
- **검증**: NL 요청→블록 생성 정확도 / 라이브 paper 1회

### 범위 밖 (비전 P3 보류)
포트폴리오 최적화 QP · 옵션 비선형(§10.1).
